# CodeX 零基础入门教学（从 0 开始）

> 面向：第一次接触 CodeX、几乎不懂终端和开发流程的新手。

---

## 1. CodeX 是什么？能帮你做什么？

你可以把 CodeX 理解成一个“会写代码、会读项目、会执行命令、还能解释过程”的 AI 开发助手。

它适合做这些工作：

1. **写代码**：根据你的描述生成功能代码。
2. **改代码**：在现有项目里定位并修改文件。
3. **查问题**：阅读报错、分析原因、给修复方案。
4. **补文档**：生成 README、接口文档、部署说明。
5. **自动化重复工作**：批量改名、格式整理、脚本化流程。
6. **教学辅导**：把复杂开发步骤翻译成新手能看懂的操作清单。

不太适合完全交给它的场景：

- 涉及法律、财务、医疗等高风险决策（需要专业人士复核）。
- 生产环境关键改动但你不做测试与评审。

---

## 2. 你现在看到的“操作页面”怎么理解？

通常可以分成几块：

1. **对话区**：你输入需求、CodeX 回复结果。
2. **命令执行区（终端）**：CodeX 会运行命令查看项目、测试代码。
3. **文件区**：显示项目文件和改动内容。
4. **结果区**：如测试输出、构建结果、截图、日志。

新手先记住一句话：

> 你只要把目标说清楚，CodeX 会尽量把“要执行的命令 + 修改的文件 + 测试结果”都给你。

## 3. 第一次使用：最稳妥的 5 步流程

### 第 1 步：说清楚“目标 + 约束 + 输出格式”

不要只说“帮我写个功能”，建议这样说：

- 目标：我要一个登录页面。
- 技术：React + TypeScript。
- 约束：只改 `src/pages` 下文件。
- 输出：先给计划，再改代码，最后跑测试。

### 第 2 步：让 CodeX 先给执行计划

你可以说：

- “先不要改代码，先给我 3~5 步计划。”

这样你先确认方向，再让它动手。

### 第 3 步：允许它执行并查看改动

让它开始改后，重点看三件事：

1. 改了哪些文件。
2. 改动是否符合你的限制。
3. 有没有跑测试或构建。

### 第 4 步：让它解释“为什么这么改”

新手非常建议追加一句：

- “请用新手能懂的话，解释每个文件改动的作用。”

### 第 5 步：做最终验收

验收至少包含：

- 功能是否达到需求。
- 命令是否成功（测试/构建）。
- 是否有潜在风险与下一步建议。

## 4. 新手最常用的提问模板（直接复制可用）

### 模板 A：做新功能

```text
你是我的编程教练。请在不改动 xxx 目录以外文件的前提下，完成以下功能：
1) ...
2) ...
请先给计划，再执行。
执行后请列出：
- 修改了哪些文件
- 每个文件改了什么
- 运行了哪些测试和结果
```

### 模板 B：修 Bug

```text
我遇到报错：<粘贴报错>
请按下面流程处理：
1) 先判断根因（给出2-3个可能性并排序）
2) 提供最小修复方案
3) 修改后运行相关测试
4) 用新手语言解释修复原理
```

### 模板 C：学习某个技术

```text
请把 <技术名> 当作零基础课程来教我：
- 先讲概念（用类比）
- 再给最小可运行例子
- 再给我两个练习题
- 最后告诉我常见错误
```

## 5. 如何“高质量下指令”（这是效果差异关键）

高质量指令 = **背景 + 目标 + 限制 + 验收标准**。

示例：

- 背景：这是电商后台项目。
- 目标：新增商品列表分页。
- 限制：不改数据库结构，不引入新依赖。
- 验收：`npm test` 通过，页面可切换页码。

如果你只说“改一下这个功能”，它容易猜错方向。

---

## 6. 新手常见误区（避免踩坑）

1. **不看改动直接合并**：一定要让它列出修改文件。
2. **不跑测试**：哪怕是最小测试也要跑。
3. **问题描述太短**：最少给“现象 + 期望 + 报错”。
4. **一次提太多需求**：建议拆成小任务，逐步完成。
5. **把敏感信息直接贴上去**：密钥、密码、个人数据要脱敏。

---

## 7. 适合哪些岗位/工作场景？

1. **前端开发**：页面、交互、组件重构。
2. **后端开发**：接口、参数校验、日志优化。
3. **测试工程**：生成测试样例、补自动化脚本。
4. **数据/脚本岗位**：写清洗脚本、批处理任务。
5. **运维/平台**：编写部署脚本、排查常见错误。
6. **产品/项目经理**（轻技术）：快速生成技术方案初稿。

## 8. 推荐的“新手学习路线”（7 天）

### Day 1-2：认识基本操作

- 学会提需求模板。
- 学会看“改了哪些文件”。
- 学会看命令结果成功/失败。

### Day 3-4：学会小步迭代

- 一次只让它做一个小功能。
- 每步都让它解释“为什么”。

### Day 5-6：学会自检

- 要求它给边界情况。
- 要求它补测试。

### Day 7：做一个完整小项目

- 需求 -> 计划 -> 编码 -> 测试 -> 复盘。

## 10. 最后给新手的建议

- 把 CodeX 当“副驾驶”，不是“自动驾驶”。
- 你负责目标和验收，CodeX 负责加速执行。
- 先学会提问结构，再追求复杂功能。

如果你愿意，我可以下一步直接给你做一份：

1. 你的专属提问模板（按你的岗位定制）
2. 一个 30 分钟上手练习（带答案）
3. 一套“看不懂报错时”的急救流程

## 11. 新手高频问答（你刚好问到的两个关键问题）

### Q1：在这个环境里用 CodeX，和直接在 GPT 对话里有什么区别？

可以把它理解成：

- **普通 GPT 对话**：偏“建议/讲解”。你通常拿到的是文字方案和示例代码。
- **CodeX 代理环境**：偏“可执行开发”。除了回答，它还能在项目里实际做事，比如：
  1) 读取仓库文件
  2) 修改文件
  3) 运行终端命令（测试、构建、检查）
  4) 生成提交（commit）和 PR 描述

一句话：**GPT 更像顾问，CodeX 更像能进项目动手的工程助手**。

### Q2：我的任何提问，都会改变最后生成的这个 `.md` 文件吗？

**不会。** 只有当你的提问明确要求“修改文档/更新文件”，或者你同意它执行改动时，文件才会被更改。

你可以用下面这三种说法控制它：

1. **只讨论，不改文件**：
   - “先只讲解方案，不要修改任何文件。”
2. **先计划，后决定改不改**：
   - “先给改动计划和将修改的文件列表，等我确认再执行。”
3. **允许执行改动**：
   - “按这个方案直接修改，并在最后列出所有改动文件和测试结果。”

实操建议（新手必用）：

- 每次都加一句：**“未经我确认，不要改文件。”**
- 执行后要求它输出：**改了哪些文件 + 每个文件改了什么 + 测试结果**。
- 如果你只想学习：明确写 **“教学模式，不落盘修改”**。

## 12. 新手再问：这份文件最后能保存在 GitHub 里吗？

**可以。** 只要你把它提交到 Git 仓库并推送到 GitHub，这个 `.md` 文件就会长期保存在仓库历史里。

你可以把流程理解成 4 步：

1. **本地修改文件**（例如 `CODEX_BEGINNER_GUIDE_CN.md`）
2. **提交 commit**（把这次改动记录到 Git 历史）
3. **推送到 GitHub**（`git push`）
4. **创建/合并 PR**（让团队可评审、可追踪）

常见判断方式：

- 在 GitHub 仓库页面能看到这个文件。
- 在提交历史（Commits）中能看到新增/修改记录。
- 在 PR 的 Files changed 中能看到具体改动。

注意：

- 如果你只是在对话里讨论、没有执行提交和推送，那它不会自动出现在 GitHub。
- 只有进入了 Git 提交历史并被推送到远端仓库，才算“真正保存在 GitHub”。
